# Spring Review Answers

## 1. Spring Security

Spring Security — это модуль Spring Framework, предназначенный для обеспечения безопасности приложения. Он решает задачи аутентификации пользователей, авторизации, защиты HTTP-запросов, управления сессиями, обработки logout, защиты от CSRF-атак и интеграции с различными механизмами входа.

Основная идея Spring Security заключается в том, что безопасность выносится в отдельный инфраструктурный слой. За счёт этого правила доступа, логика входа и ограничения на выполнение операций не смешиваются с бизнес-логикой приложения.

Архитектурно Spring Security работает через цепочку фильтров, которые обрабатывают каждый HTTP-запрос до попадания его в контроллер. В ходе этой обработки система:
- извлекает данные пользователя из запроса;
- выполняет аутентификацию;
- помещает результат в контекст безопасности;
- проверяет, имеет ли пользователь право на доступ к ресурсу.

Spring Security поддерживает разные способы аутентификации:
- form login;
- HTTP Basic;
- JWT;
- OAuth2;
- LDAP;
- пользовательские механизмы входа.

Таким образом, Spring Security представляет собой полноценную инфраструктуру безопасности для Spring-приложений, а не просто набор аннотаций для ограничения доступа.

## 2. Что такое авторизация, аутентификация

Аутентификация — это процесс установления личности пользователя. Она отвечает на вопрос: кто именно обращается к системе. В рамках аутентификации приложение проверяет предоставленные учетные данные: логин и пароль, токен, сертификат, одноразовый код и другие варианты идентификации.

Авторизация — это процесс проверки прав доступа уже аутентифицированного пользователя. Она отвечает на вопрос: какие действия разрешены пользователю после того, как его личность подтверждена.

Различие между этими понятиями принципиальное:
- аутентификация подтверждает личность;
- авторизация определяет права доступа.

Пример:
- пользователь ввёл логин и пароль;
- система проверила, что это действительно пользователь `ivan`;
- затем система определяет, может ли `ivan` открыть административную страницу, удалить запись или изменить данные другого пользователя.

В HTTP-контексте эти процессы часто соотносятся со статусами ответа:
- `401 Unauthorized` обычно указывает на отсутствие или некорректность аутентификации;
- `403 Forbidden` означает, что пользователь распознан, но прав доступа недостаточно.

## 3. Объекты Principal, Authorities, Authentication

В Spring Security используются несколько ключевых объектов, описывающих состояние безопасности текущего пользователя.

`Principal` — это объект, представляющий текущего пользователя. В простейшем случае он содержит имя пользователя, однако на практике в качестве principal часто используется объект `UserDetails` или пользовательский объект с более широким набором данных.

`Authorities` — это права пользователя. Технически они представлены как коллекция `GrantedAuthority`. В состав authorities могут входить:
- роли, например `ROLE_USER`, `ROLE_ADMIN`;
- отдельные разрешения, например `READ_REPORTS`, `WRITE_USERS`.

На основании authorities система определяет, какие действия доступны пользователю.

`Authentication` — это центральный объект Spring Security, содержащий полную информацию о текущем состоянии аутентификации. В него входят:
- `principal` — пользователь;
- `credentials` — учетные данные;
- `authorities` — права пользователя;
- `authenticated` — признак успешной аутентификации.

После успешной аутентификации объект `Authentication` сохраняется в `SecurityContext`. Далее он используется при обработке запросов, проверке ролей, работе аннотаций безопасности и получении информации о текущем пользователе в контроллерах и сервисах.

Связь между этими объектами следующая:
- `Principal` показывает, кто пользователь;
- `Authorities` показывают, что ему разрешено;
- `Authentication` объединяет эту информацию в единый объект безопасности.

## 4. Чем отличается InMemoryAuthentication от basicAuthentication

`InMemoryAuthentication` и `Basic Authentication` относятся к разным уровням системы безопасности.

`InMemoryAuthentication` — это способ хранения пользователей и их ролей внутри приложения, в оперативной памяти. Пользователи задаются в конфигурации и существуют только в процессе работы приложения. После перезапуска они создаются заново.

Такой подход применяется:
- в учебных проектах;
- в тестовых стендах;
- в демонстрационных приложениях;
- для быстрого прототипирования.

Особенности `InMemoryAuthentication`:
- не требует базы данных;
- прост в настройке;
- не подходит для полноценного production-использования;
- неудобен для динамического управления пользователями.

`Basic Authentication` — это способ передачи учетных данных по HTTP. Клиент отправляет логин и пароль в заголовке `Authorization` по схеме `Basic`. Сервер получает эти данные и пытается аутентифицировать пользователя.

Таким образом:
- `InMemoryAuthentication` отвечает за источник пользователей;
- `Basic Authentication` отвечает за механизм передачи логина и пароля в запросе.

Они могут использоваться совместно. Например, пользователь может передавать логин и пароль через HTTP Basic, а проверка этих данных будет происходить по списку пользователей, хранящихся in-memory.

Следовательно, это не взаимоисключающие понятия, а разные элементы одной системы:
- одно определяет, где хранятся пользователи;
- другое определяет, как выполняется вход.

## 5. Как мы можем добавить секьюрность к контроллеру?

Безопасность к контроллеру в Spring можно добавить несколькими способами.

Первый способ — ограничение доступа на уровне HTTP-запросов через конфигурацию `SecurityFilterChain`. В этом случае правила задаются по URL-шаблонам. Например, можно указать, что `/admin/**` доступен только пользователям с ролью `ADMIN`, а `/user/**` — пользователям с ролями `USER` или `ADMIN`.

Этот подход подходит для централизованного управления доступом ко всем маршрутам приложения. Его преимущество состоит в том, что правила безопасности собраны в одном месте.

Второй способ — использование аннотаций на уровне методов контроллера или сервисного слоя:
- `@PreAuthorize`
- `@PostAuthorize`
- `@Secured`
- `@RolesAllowed`

Такой подход позволяет задавать ограничения непосредственно рядом с бизнес-операцией. Например, метод можно разрешить только пользователям с ролью `ADMIN` или проверить доступ через выражение SpEL.

Дополнительно безопасность может быть реализована:
- через сервисный слой, а не только через контроллер;
- через кастомные фильтры;
- через JWT-аутентификацию;
- через анализ текущего пользователя с помощью `@AuthenticationPrincipal`.

Наиболее распространённые способы:
- конфигурация доступа по URL через `SecurityFilterChain`;
- ограничение доступа через аннотации method security.

## 6. Связи таблиц many-to-many, one-to-one

Связь `one-to-one` означает, что одной записи из первой таблицы соответствует одна запись из второй таблицы. Такая связь используется в случаях, когда объект логически разделён на две сущности, но между ними сохраняется строгое соответствие один к одному.

Примеры:
- пользователь и профиль;
- человек и паспорт;
- заказ и платёжная информация.

На уровне базы данных `one-to-one` обычно реализуется:
- внешним ключом с ограничением `UNIQUE`;
- либо общей первичной колонкой.

В JPA такая связь описывается аннотацией `@OneToOne`. Одна из сторон является владельцем связи и содержит `@JoinColumn`, то есть именно она управляет внешним ключом.

Связь `many-to-many` означает, что одной записи из первой таблицы может соответствовать множество записей из второй таблицы, и наоборот.

Примеры:
- студент и курс;
- пользователь и роль;
- товар и категория.

На уровне реляционной базы данных связь `many-to-many` напрямую не хранится. Для неё создаётся промежуточная таблица, которая содержит внешние ключи на обе связанные таблицы.

В JPA такая связь описывается через `@ManyToMany` и, как правило, `@JoinTable`.

Особенность связи `many-to-many` заключается в том, что она удобна только в простых случаях. Если у самой связи появляются дополнительные поля, например:
- дата назначения;
- статус;
- источник;
- приоритет;
- срок действия,

то вместо `many-to-many` обычно создают отдельную сущность связи. В этом случае отношение раскладывается на две связи:
- `one-to-many`;
- `many-to-one`.

Такой подход лучше соответствует реальной структуре БД и даёт больше контроля над моделью данных.

## 7. Как работают каскады для таблиц и какие они бывают

Каскады в JPA/Hibernate определяют, какие операции над родительской сущностью автоматически распространяются на связанные сущности.

Если между сущностями установлена связь и для неё задан каскад, то при выполнении операции над родителем ORM может выполнить аналогичную операцию над дочерними объектами.

Основные типы каскадов:
- `PERSIST` — при сохранении родительской сущности сохраняются связанные дочерние сущности;
- `MERGE` — при обновлении родительской сущности выполняется merge дочерних сущностей;
- `REMOVE` — при удалении родительской сущности удаляются связанные дочерние сущности;
- `REFRESH` — дочерние сущности обновляются из базы данных вместе с родительской;
- `DETACH` — дочерние сущности отсоединяются от persistence context;
- `ALL` — включает все перечисленные типы каскадов.

Каскады удобны в тех случаях, когда дочерний объект логически принадлежит родителю и не должен существовать отдельно. Например:
- заказ и позиции заказа;
- корзина и элементы корзины;
- документ и его вложенные строки.

При использовании каскадов требуется аккуратность. Каскадное удаление не должно применяться к сущностям, которые могут использоваться в других частях системы. Например, удаление роли, справочника или пользователя вместе с другой сущностью может привести к потере данных или нарушению ссылочной целостности.

Отдельно различают:
- `CascadeType.REMOVE` — удаление дочерних сущностей при удалении родителя;
- `orphanRemoval = true` — удаление дочернего объекта, если он был исключён из коллекции родителя и больше не связан с ним.

Следовательно, каскады управляют распространением операций жизненного цикла сущности на связанные объекты и используются там, где между сущностями существует отношение владения.

## 8. Bootstrap

Bootstrap — это frontend-фреймворк для разработки пользовательских интерфейсов. Он содержит готовые CSS-классы, адаптивную сетку, стандартные UI-компоненты и JavaScript-механизмы для типовых элементов интерфейса.

Bootstrap предназначен для ускорения разработки веб-интерфейсов. Он предоставляет:
- grid system для построения адаптивной разметки;
- готовые стили кнопок, форм, таблиц, карточек, навигации;
- набор утилитарных классов;
- JavaScript-компоненты для modal, dropdown, tabs, collapse и других элементов.

Одной из ключевых особенностей Bootstrap является адаптивность. Сетка на 12 колонок позволяет строить интерфейсы, которые корректно отображаются на разных размерах экрана без значительного объёма ручной вёрстки.

Преимущества Bootstrap:
- высокая скорость разработки;
- большое количество готовых компонентов;
- единообразный внешний вид интерфейса;
- хорошая документация;
- удобство для прототипирования и административных панелей.

Недостатки Bootstrap:
- типовой внешний вид, из-за которого интерфейсы часто выглядят одинаково;
- ограниченная гибкость при создании уникального дизайна;
- наличие лишних стилей, если используется только часть возможностей фреймворка;
- зависимость от готовых классов вместо построения собственной дизайн-системы.

Bootstrap широко используется для быстрого создания адаптивных интерфейсов, особенно в проектах, где важны скорость разработки и предсказуемый результат.

## 9. Rest-сервисы. Их преимущества и недостатки

REST (*Representational State Transfer*) — это архитектурный стиль построения распределённых приложений, в котором взаимодействие между клиентом и сервером организуется через HTTP и представление ресурсов. В REST основной сущностью считается ресурс, который имеет уникальный URI. Клиент обращается к ресурсу по URL и выполняет над ним стандартные HTTP-операции: получение, создание, изменение, удаление.

В REST-сервисах обычно используются следующие принципы:
- ресурсная модель, где каждая сущность доступна по своему URI;
- использование стандартных HTTP-методов (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`);
- stateless-подход, при котором сервер не хранит состояние клиента между запросами;
- единообразный интерфейс взаимодействия;
- использование стандартных кодов ответа HTTP.

Пример ресурсного подхода:
- `GET /users` — получить список пользователей;
- `GET /users/10` — получить пользователя с id `10`;
- `POST /users` — создать нового пользователя;
- `PUT /users/10` — полностью обновить пользователя;
- `DELETE /users/10` — удалить пользователя.

Преимущества REST:
- простота и понятность архитектуры;
- использование стандартного протокола HTTP;
- хорошая совместимость между разными системами и платформами;
- удобство интеграции frontend, mobile и backend-приложений;
- независимость клиента и сервера;
- масштабируемость за счёт stateless-подхода;
- широкая поддержка в инструментах, библиотеках и фреймворках.

Недостатки REST:
- отсутствие строгого стандарта структуры ответа, из-за чего разные API могут быть организованы по-разному;
- возможная избыточность данных в ответах;
- необходимость нескольких запросов для получения сложных связанных данных;
- не всегда удобен для сценариев с высокой динамикой, realtime-взаимодействием и сложными вложенными выборками;
- вопросы версионирования API требуют отдельной стратегии.

REST широко используется в современных веб-приложениях, микросервисной архитектуре и интеграционных системах благодаря своей простоте, предсказуемости и тесной связи с HTTP.

## 10. Форматы данных, использующиеся в REST-сервисах

REST не привязан к одному конкретному формату представления данных. Он определяет архитектурный стиль взаимодействия, а формат полезной нагрузки может быть различным.

Наиболее распространённые форматы данных в REST-сервисах:
- `JSON`;
- `XML`;
- `plain text`;
- `HTML`;
- бинарные данные, например файлы;
- реже `YAML` или специализированные форматы.

На практике основным форматом является `JSON` (*JavaScript Object Notation*). Он получил широкое распространение по нескольким причинам:
- компактный и легко читаемый;
- хорошо поддерживается браузерами и JavaScript;
- просто сериализуется и десериализуется;
- широко поддерживается backend-фреймворками, в том числе Spring через Jackson.

Пример JSON:
```json
{
  "id": 1,
  "name": "Ivan",
  "role": "USER"
}
```

`XML` также используется в REST-сервисах, особенно в старых корпоративных интеграциях. Его преимущества — строгая структура и развитая экосистема схем и валидации. Недостатки — больший объём и более высокая сложность обработки по сравнению с JSON.

Формат ответа и запроса определяется через HTTP-заголовки:
- `Content-Type` — показывает, в каком формате отправляется тело запроса;
- `Accept` — показывает, какой формат ответа ожидает клиент.

Примеры:
- `Content-Type: application/json`
- `Accept: application/xml`

Таким образом, REST-сервисы чаще всего используют JSON, но могут работать и с другими форматами, если это требуется архитектурой или интеграционными требованиями.

## 11. Что такое `@ResponseBody`, `@RequestBody`, `ResponseEntity`

`@RequestBody` используется для чтения тела HTTP-запроса и преобразования его в Java-объект. Когда клиент отправляет JSON или XML в теле запроса, Spring через `HttpMessageConverter` десериализует эти данные в объект соответствующего типа.

Пример:
```java
@PostMapping("/users")
public User createUser(@RequestBody User user) {
    return userService.save(user);
}
```

В данном случае JSON из тела запроса будет преобразован в объект `User`.

`@ResponseBody` указывает Spring, что возвращаемое методом значение нужно не интерпретировать как имя представления, а записать непосредственно в тело HTTP-ответа. Если метод возвращает объект, Spring сериализует его в подходящий формат, чаще всего в JSON.

Пример:
```java
@GetMapping("/users/{id}")
@ResponseBody
public User getUser(@PathVariable Long id) {
    return userService.findById(id);
}
```

Без `@ResponseBody` в обычном `@Controller` строка или объект могли бы быть обработаны через механизм view resolution.

`ResponseEntity` — это специальный класс Spring, представляющий полный HTTP-ответ. Он позволяет управлять:
- телом ответа;
- статусом ответа;
- HTTP-заголовками.

Пример:
```java
@GetMapping("/users/{id}")
public ResponseEntity<User> getUser(@PathVariable Long id) {
    User user = userService.findById(id);
    return ResponseEntity.ok(user);
}
```

Пример с обработкой отсутствующего объекта:
```java
@GetMapping("/users/{id}")
public ResponseEntity<User> getUser(@PathVariable Long id) {
    User user = userService.findById(id);
    if (user == null) {
        return ResponseEntity.notFound().build();
    }
    return ResponseEntity.ok(user);
}
```

`ResponseEntity` применяется в тех случаях, когда требуется точный контроль над HTTP-ответом. Например:
- вернуть `201 Created` после создания ресурса;
- вернуть `204 No Content`, если тело ответа не требуется;
- вернуть `404 Not Found`, если ресурс отсутствует;
- добавить кастомные заголовки.

Таким образом:
- `@RequestBody` читает тело запроса;
- `@ResponseBody` записывает результат метода в тело ответа;
- `ResponseEntity` позволяет полностью управлять структурой HTTP-ответа.

## 12. Что такое AJAX / fetch

AJAX (*Asynchronous JavaScript and XML*) — это подход к взаимодействию браузера с сервером, при котором запросы выполняются асинхронно, без полной перезагрузки страницы. Это позволяет динамически подгружать данные, отправлять формы, обновлять элементы интерфейса и строить более интерактивные веб-приложения.

Исторически AJAX реализовывался через объект `XMLHttpRequest`. Несмотря на название, обмен мог происходить не только в XML, но и в JSON, plain text и других форматах.

Современный подход чаще связан с использованием `fetch`. `fetch` — это встроенный JavaScript API для выполнения HTTP-запросов. Он является более современным, удобным и читаемым инструментом по сравнению с `XMLHttpRequest`.

Пример:
```javascript
fetch('/api/users')
  .then(response => response.json())
  .then(data => console.log(data));
```

`fetch` позволяет:
- отправлять `GET`, `POST`, `PUT`, `DELETE` и другие HTTP-запросы;
- передавать заголовки;
- отправлять тело запроса;
- получать и обрабатывать ответ сервера;
- работать с промисами и асинхронным кодом.

Пример POST-запроса:
```javascript
fetch('/api/users', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    name: 'Ivan'
  })
});
```

Основные преимущества AJAX/fetch:
- отсутствие полной перезагрузки страницы;
- улучшение пользовательского опыта;
- возможность динамически обновлять данные;
- удобная интеграция с REST API.

AJAX — это общий подход асинхронного обмена данными с сервером, а `fetch` — один из современных инструментов реализации этого подхода.

## 13. Чем аннотация `@RestController` отличается от `@Controller`

`@Controller` — это базовая аннотация Spring MVC для контроллеров. Она помечает класс как компонент, который обрабатывает HTTP-запросы. По умолчанию методы такого контроллера возвращают:
- имя представления;
- `ModelAndView`;
- либо данные, если дополнительно используется `@ResponseBody`.

Пример:
```java
@Controller
public class PageController {

    @GetMapping("/home")
    public String home() {
        return "home";
    }
}
```

В этом случае строка `"home"` интерпретируется как имя представления, которое должно быть найдено через `ViewResolver`.

`@RestController` — это специализированная аннотация, которая объединяет:
- `@Controller`
- `@ResponseBody`

Это означает, что все методы класса автоматически возвращают данные в тело HTTP-ответа. Такие контроллеры чаще всего используются для построения REST API.

Пример:
```java
@RestController
public class UserController {

    @GetMapping("/users")
    public List<User> getUsers() {
        return userService.findAll();
    }
}
```

В этом случае список пользователей будет сериализован в JSON и отправлен клиенту.

Основное различие:
- `@Controller` чаще применяется для MVC-приложений, где возвращаются страницы;
- `@RestController` применяется для REST-сервисов, где возвращаются данные.

Следовательно, если задача контроллера — отдавать HTML-представления, обычно используется `@Controller`. Если задача — отдавать JSON или XML в ответ на HTTP-запросы, используется `@RestController`.

## 14. RestTemplate и его методы

`RestTemplate` — это HTTP-клиент Spring для взаимодействия с внешними REST-сервисами. Он позволяет приложению отправлять HTTP-запросы к другим сервисам и получать ответы в виде Java-объектов.

`RestTemplate` долгое время был основным инструментом Spring для синхронной работы с HTTP. В современных версиях Spring для новых проектов часто рекомендуют `WebClient`, однако `RestTemplate` по-прежнему широко встречается в существующих приложениях.

Основные методы `RestTemplate`:

`getForObject()`  
Выполняет `GET`-запрос и возвращает только тело ответа, преобразованное в указанный тип.

Пример:
```java
User user = restTemplate.getForObject(url, User.class);
```

`getForEntity()`  
Выполняет `GET`-запрос и возвращает `ResponseEntity`, то есть полный HTTP-ответ со статусом, заголовками и телом.

Пример:
```java
ResponseEntity<User> response = restTemplate.getForEntity(url, User.class);
```

`postForObject()`  
Выполняет `POST`-запрос и возвращает тело ответа.

Пример:
```java
User created = restTemplate.postForObject(url, requestBody, User.class);
```

`postForEntity()`  
Выполняет `POST`-запрос и возвращает полный объект `ResponseEntity`.

`put()`  
Выполняет `PUT`-запрос. Обычно используется для полного обновления ресурса. Возвращаемого тела, как правило, не предполагает.

Пример:
```java
restTemplate.put(url, requestBody);
```

`delete()`  
Выполняет `DELETE`-запрос.

Пример:
```java
restTemplate.delete(url);
```

`exchange()`  
Наиболее универсальный метод. Позволяет указать:
- HTTP-метод;
- заголовки;
- тело запроса;
- тип ответа.

Пример:
```java
HttpHeaders headers = new HttpHeaders();
headers.set("Authorization", "Bearer token");

HttpEntity<String> entity = new HttpEntity<>(headers);

ResponseEntity<String> response = restTemplate.exchange(
    url,
    HttpMethod.GET,
    entity,
    String.class
);
```

`patchForObject()`  
Используется для `PATCH`-запросов, если HTTP-клиент поддерживает этот метод.

Дополнительные методы:
- `headForHeaders()` — выполняет `HEAD` и возвращает заголовки;
- `optionsForAllow()` — возвращает набор разрешённых HTTP-методов для ресурса.

`RestTemplate` часто используется для:
- вызова внешних REST API;
- интеграции с другими микросервисами;
- обмена данными между backend-приложениями;
- отправки запросов в сторонние системы.

## 15. HTTP-протокол

HTTP (*HyperText Transfer Protocol*) — это прикладной протокол передачи данных, используемый для взаимодействия между клиентом и сервером. Он лежит в основе работы веб-приложений, REST-сервисов и большинства интернет-API.

HTTP работает по модели `request-response`:
- клиент отправляет запрос;
- сервер обрабатывает его;
- сервер возвращает ответ.

HTTP является stateless-протоколом, то есть каждый запрос считается независимым, и сервер не обязан помнить состояние клиента между запросами. Для хранения пользовательского состояния обычно используют:
- cookies;
- session;
- токены;
- внешние хранилища состояния.

Структура HTTP-запроса включает:
- метод;
- URI;
- версию протокола;
- заголовки;
- тело запроса.

Пример:
```http
POST /users HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "name": "Ivan"
}
```

Структура HTTP-ответа включает:
- статус-код;
- заголовки;
- тело ответа.

Пример:
```http
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 1,
  "name": "Ivan"
}
```

Основные HTTP-методы:
- `GET` — получение ресурса;
- `POST` — создание ресурса или выполнение операции;
- `PUT` — полное обновление ресурса;
- `PATCH` — частичное обновление ресурса;
- `DELETE` — удаление ресурса;
- `HEAD` — получение только заголовков;
- `OPTIONS` — получение информации о доступных методах.

Группы HTTP-статусов:
- `1xx` — информационные;
- `2xx` — успешное выполнение;
- `3xx` — перенаправления;
- `4xx` — ошибки клиента;
- `5xx` — ошибки сервера.

Наиболее часто используемые коды:
- `200 OK` — успешный запрос;
- `201 Created` — ресурс успешно создан;
- `204 No Content` — успешный запрос без тела ответа;
- `400 Bad Request` — некорректный запрос;
- `401 Unauthorized` — пользователь не аутентифицирован;
- `403 Forbidden` — доступ запрещён;
- `404 Not Found` — ресурс не найден;
- `500 Internal Server Error` — внутренняя ошибка сервера.

Большое значение в HTTP имеют заголовки. Они передают служебную информацию, например:
- формат передаваемых данных;
- тип авторизации;
- cookies;
- кэширование;
- язык;
- длину содержимого.

Примеры распространённых заголовков:
- `Content-Type`
- `Accept`
- `Authorization`
- `Set-Cookie`
- `Cache-Control`

Отдельно выделяют HTTPS — это HTTP поверх TLS/SSL, то есть HTTP с шифрованием. HTTPS обеспечивает:
- конфиденциальность данных;
- защиту от перехвата;
- подтверждение подлинности сервера;
- целостность передаваемой информации.

HTTP-протокол является базой для работы Spring MVC и REST API. Контроллеры обрабатывают HTTP-запросы, тело запросов преобразуется в объекты, а результаты выполнения методов формируют HTTP-ответы со статусами, заголовками и телом.
